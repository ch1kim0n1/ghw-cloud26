# Data Schema Definitions (SQLite MVP + Postgres Migration Path)

## Document Purpose
Define all core data entities, their relationships, and persistence requirements for the MVP system. This document provides the contract for database design (SQLite for MVP, Postgres migration path for production).

## 1. Architecture Assumptions
- **MVP Database:** SQLite (lightweight, local development)
- **Production Path:** PostgreSQL (scalable, JSONB support)
- **ORM/Driver:** Any Go SQL library (database/sql + sqlite3 driver for MVP)
- **Schema Versioning:** Migrations tracked in `scripts/migrations/`
- **No ORMs in MVP:** Direct SQL for control + simplicity (one developer, one week)

---

## 2. Relational Schema (SQL-Oriented)

### Table: campaigns
```sql
CREATE TABLE campaigns (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  product_id TEXT NOT NULL,
  video_filename TEXT NOT NULL,
  video_path TEXT NOT NULL,
  duration_seconds REAL,
  target_ad_duration_seconds INTEGER DEFAULT 5,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  
  FOREIGN KEY (product_id) REFERENCES products(id)
);
```

**Key Fields:**
- `id`: UUID-like primary key (generated by app)
- `video_path`: Local filesystem path for MVP (e.g., `/tmp/uploads/video_abc123.mp4`)
- `target_ad_duration_seconds`: User constraint (5–8 seconds)

---

### Table: products
```sql
CREATE TABLE products (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  description TEXT,
  image_path TEXT,
  category TEXT,
  context_keywords TEXT,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  
  UNIQUE (name, category)
);
```

**Key Fields:**
- `context_keywords`: Comma-separated or JSON tags for relevance scoring
- `image_path`: Local filesystem path to product asset

---

### Table: jobs
```sql
CREATE TABLE jobs (
  id TEXT PRIMARY KEY,
  campaign_id TEXT NOT NULL,
  status TEXT DEFAULT 'queued',
  progress_percent INTEGER DEFAULT 0,
  current_stage TEXT,
  error_message TEXT,
  result_video_path TEXT,
  metadata_json TEXT,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  started_at TIMESTAMP,
  completed_at TIMESTAMP,
  
  FOREIGN KEY (campaign_id) REFERENCES campaigns(id),
  INDEX (status),
  INDEX (created_at DESC)
);
```

**Status Machine:**
```
queued → analyzing → generating → stitching → completed (or failed)
```

**Key Fields:**
- `status`: One of: queued, analyzing, generating, stitching, completed, failed
- `current_stage`: Granular sub-stage (scene_detection, context_analysis, slot_ranking, ad_generation, rendering)
- `metadata_json`: Stores job context as JSON (e.g., selected_slot_id, generation_metrics)

**Example `metadata_json`:**
```json
{
  "selected_slot_id": "slot_xyz789",
  "selected_slot_rank": 1,
  "generation_model": "RIFE",
  "generation_service": "Replicate",
  "replicate_prediction_id": "pred_abc123",
  "total_scenes": 24,
  "top_3_slots": [
    { "rank": 1, "score": 0.92 },
    { "rank": 2, "score": 0.87 },
    { "rank": 3, "score": 0.82 }
  ]
}
```

---

### Table: scenes
```sql
CREATE TABLE scenes (
  id TEXT PRIMARY KEY,
  job_id TEXT NOT NULL,
  scene_number INTEGER,
  start_frame INTEGER,
  end_frame INTEGER,
  duration_seconds REAL,
  
  -- Context features (populated by context analysis)
  motion_score REAL,
  stability_score REAL,
  dialogue_present BOOLEAN,
  dialogue_gap_start_frame INTEGER,
  dialogue_gap_end_frame INTEGER,
  scene_description TEXT,
  
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  
  FOREIGN KEY (job_id) REFERENCES jobs(id),
  INDEX (job_id),
  INDEX (scene_number)
);
```

**Key Fields:**
- `motion_score`, `stability_score`: 0-1 range, used for ranking (low motion = good insertion point)
- `dialogue_gap_*`: Frame numbers identifying quiet moments
- `scene_description`: AI summary for product/context matching

---

### Table: slots
```sql
CREATE TABLE slots (
  id TEXT PRIMARY KEY,
  job_id TEXT NOT NULL,
  scene_id TEXT NOT NULL,
  rank INTEGER,
  slot_type TEXT,
  insertion_frame INTEGER,
  confidence REAL,
  score REAL,
  reasoning TEXT,
  selected BOOLEAN DEFAULT 0,
  
  -- After selection: generation metadata
  generated_video_path TEXT,
  generation_status TEXT,
  generation_error TEXT,
  
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  
  FOREIGN KEY (job_id) REFERENCES jobs(id),
  FOREIGN KEY (scene_id) REFERENCES scenes(id),
  INDEX (job_id, rank),
  UNIQUE (job_id, rank)
);
```

**Key Fields:**
- `rank`: Ordering (1 = best candidate)
- `slot_type`: dialogue_gap, low_motion, scene_boundary
- `reasoning`: Human-readable explanation for judges
- `generated_video_path`: Path to RIFE-interpolated ad clip (e.g., `/tmp/outputs/slot_xyz_generated.mp4`)

---

### Table: renders
```sql
CREATE TABLE renders (
  id TEXT PRIMARY KEY,
  job_id TEXT NOT NULL,
  slot_id TEXT NOT NULL,
  output_video_path TEXT NOT NULL,
  status TEXT DEFAULT 'in_progress',
  error_message TEXT,
  metrics_json TEXT,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  completed_at TIMESTAMP,
  
  FOREIGN KEY (job_id) REFERENCES jobs(id),
  FOREIGN KEY (slot_id) REFERENCES slots(id),
  INDEX (job_id)
);
```

**Key Fields:**
- `output_video_path`: Final stitched video (source + interpolated ad) e.g., `/tmp/outputs/final_output.mp4`
- `metrics_json`: Frame counts, duration, blending quality

**Example `metrics_json`:**
```json
{
  "total_frames": 12000,
  "insertion_start_frame": 5600,
  "insertion_duration_seconds": 6.5,
  "stitching_service": "ffmpeg",
  "quality_score": 0.95
}
```

---

### Table: job_logs (Audit Trail)
```sql
CREATE TABLE job_logs (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  job_id TEXT NOT NULL,
  timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  event_type TEXT,
  stage_name TEXT,
  message TEXT,
  details_json TEXT,
  
  FOREIGN KEY (job_id) REFERENCES jobs(id),
  INDEX (job_id, timestamp DESC)
);
```

**Event Types:**
- stage_started
- stage_completed
- error
- slot_ranked
- inference_started
- inference_completed

## 4. Foreign Key Strategy

**Referential Integrity:**
- MVP: Soft deletes (add `deleted_at` column, don't hard delete)
- Production: Add `ON DELETE CASCADE` constraints for audit trail

**Null Handling:**
- `result_video_path` in `jobs`: NULL until stitching completes
- `generation_metadata_json`: populated only after generation attempt

## 5. Data Modeling Rules
- all primary business entities use UUIDs
- status fields must be enum-constrained at application layer
- metadata that evolves quickly belongs in JSONB unless it needs relational querying
- large binary/video assets never belong in the database directly
- DB stores references, not heavy media payloads
